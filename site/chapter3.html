SECTION 2: PROJECT GOVERNANCE AND MANAGEMENT (3.1)

2.1 What is Project Governance?

  • Project Governance = Structures, policies, procedures, and controls that ensure strategic and tactical alignment of projects with enterprise goals
  • Without proper governance, ALL aspects of a project may be compromised
  • Governance decisions should be driven through the business case
  • Must involve periodic review of benefits achieved

2.2 Effective Project Management Factors

Projects must be managed based on THREE types of factors:

Factor Type Examples
Hard Factors Deliverables, quality, costs, deadlines
Soft Factors Team dynamics, conflict resolution, leadership, culture, communication
Environmental Factors Political/power issues, stakeholder expectations, ethical/social issues

💡 Simple Explanation: Hard = what you deliver; Soft = how you work together; Environmental = who's watching.

2.3 Project Management Practices (3.1.1)

  • Project Management = Application of knowledge, skills, tools and techniques to achieve a stated objective
  • Begins with project charter → Ends with project completion
  • Key processes: Initiating → Planning → Executing → Controlling → Monitoring → Closing

2.4 Project Management Structures (3.1.2)

Three organizational structures determine authority within a project:

Structure PM Authority Description
Functional NONE (advisory only) Work broken into departments; PM can only advise
Project-Structured FULL authority PM has formal authority over budget, schedule and team
Matrix SHARED Authority shared between PM and department heads

📌 EXAM TIP: In a functional structure, the project manager has NO formal authority — can only advise. This is the weakest PM authority.

Key Points on IS Projects:

  • Projects should be submitted to and prioritized by the IT Steering Committee
  • The IT Steering Committee identifies the project manager
  • The PM does NOT need to be an IT staff member
  • IS Auditor included as a control expert in an advisory role (not audit role)
  • If IS Auditor is too involved, they may become ineligible to audit the completed application

2.5 Project Management Roles and Responsibilities (3.1.3)

Role Key Responsibilities
Project Steering Committee Overall direction; ultimately responsible for deliverables, costs, schedule; includes senior reps from all affected business areas; chaired by project sponsor
Senior Management Demonstrates commitment; approves necessary resources
Project Sponsor Provides funding; works with PM to define critical success factors; owns data and application; typically senior manager of primary business unit
User Management Assumes ownership of project; allocates qualified reps; participates in business process redesign, requirements, testing, training
User Project Team Completes tasks; communicates with system developers as subject matter experts
Project Manager Day-to-day management; ensures adherence to standards; resolves conflicts; monitors costs and timetable; facilitates scope definition
Quality Assurance (QA) Reviews results within/at end of each phase; confirms compliance; measures adherence to SDLC; advises on deviations
Systems Development Management Technical support; ensures system compatibility; assumes post-installation support
Security Officer Ensures system controls provide effective protection; consults on security throughout life cycle
IS Security Engineer Identifies security vulnerabilities; defines security architectures and designs

📌 EXAM TIP: The Project Sponsor is ultimately accountable for the project — they own the data and application and provide funding.

2.6 Project Management Techniques (3.1.4)

Three intertwining elements must always be considered: - Deliverables (what) - Duration (when) - Budget (how much)

Key Formula: R × D = TR (Resources × Duration = Total Resources) - This is the classic "man × month dilemma" - Fewer resources = longer project - More resources = shorter project (but with limits — LR and LD)

2.7 Portfolio/Program Management (3.1.5)

Term Definition
Project Single initiative with specific objectives, deliverables, start/end dates
Program Group of projects closely linked through common strategies, objectives, budgets, schedules
Portfolio ALL projects in an enterprise at a given point in time
PMO (Project Management Office) Permanent structure that governs project management processes; does NOT manage project content

Program vs. Project vs. Portfolio:

PORTFOLIO (all projects) ├── PROGRAM A (group of related projects) │ ├── Project A1 │ └── Project A2 ├── PROGRAM B │ └── Project B1 └── Standalone Project C

Program characteristics (vs. Projects):

  • More complex
  • Longer duration
  • Higher budget
  • Higher risk
  • Higher strategic importance

Program Roles:

  • Program Owner — distinct from project owner
  • Program Manager
  • Program Team

PMO Objectives (Project Portfolio Management):

  1. Optimization of the overall portfolio (not individual projects)
  2. Prioritizing and scheduling projects
  3. Resource coordination (internal and external)
  4. Knowledge transfer throughout projects

Project Portfolio Database (Mandatory):

Must include: owner, schedules, objectives, project type, status, cost Typical reports: bar chart, profit-vs-risk matrix, progress graph

2.8 Project Benefits Realization (3.1.7)

Benefits Realization = Ensuring IT investments deliver measurable business value

Key Elements:

  1. Describe benefits management
  2. Assign a measure and target
  3. Establish tracking/measuring regimen
  4. Document assumptions
  5. Establish key responsibilities
  6. Validate benefits predicted
  7. Plan the benefit to be realized

Four Phases of Benefits Realization:

  1. Planning — Plan the benefits
  2. Execution — Deliver the project
  3. Transition — Move to production
  4. Operations — Realize and measure benefits

💡 Business Example (ROI Calculation): - Costs: Development + controls for integrity - Benefits: Increase in operating profits + reduction in operating costs - ROI = Value of benefit / Costs → compare to enterprise cost of funds (go/no-go)

📌 If an enterprise consistently fails to meet ROI objectives → suggests weakness in SDLC and project management

2.9 Project Initiation (3.1.8)

Project Charter = Document stating objective, stakeholders, PM and sponsor PID (Project Initiation Document) or PRD (Project Request Document) = Authorization for project to begin

Methods of Project Initiation:

Method Description
One-on-one meetings Two-way communication between team members and PM
Kick-off meetings PM informs team of what must be done
Project start workshops Preferred method; ensures open communication; builds common overview
Combination Two or more methods combined

2.10 Project Objectives (3.1.9)

  • Project objectives must be SMART: Specific, Measurable, Attainable, Realistic, Timely

OBS vs. WBS:

| OBS (Object Breakdown Structure) | WBS (Work Breakdown Structure) |---|---|---| What it shows | Individual components of the solution and their relationships | Tasks necessary to build each OBS element | Focus | What will be built (deliverables) | How it will be built (work packages) | Format | Hierarchical (graphical or table) | Process-oriented and phased |

Work Packages (WPs) — Key Rules:

  • Each WP must have a distinct owner
  • WPs must include dependencies on other WPs
  • WPs should NOT exceed 10 days in duration
  • WPs must be independent of each other
  • WPs must be unique (not duplicated across the WBS)
  • Top WBS level = final deliverable/project
  • Subdeliverables contain WPs assigned to departments

2.11 Project Planning (3.1.10)

IS Development Project Cost Estimation Methods:

Method Speed Accuracy Description
Analogous Estimating Fastest Lowest Uses estimates from prior projects
Parametric Estimating Fast Good Uses statistical data from past projects (hours, material costs)
Bottom-Up Estimating Slowest Highest Estimates each activity detail; sums all up
Actual Costs Fast Good Extrapolates from actual costs of same system in past

Function Point Analysis (FPA):

  • FPA = Multiple-point technique for estimating complexity in large business applications
  • Measures size based on inputs, outputs, files, interfaces, and queries
  • 5 FP count values: 1. Number of user inputs 2. Number of user outputs 3. Number of user inquiries 4. Number of files 5. Number of external interfaces
Parameter Simple Average Complex
User inputs 3 4 6
User outputs 4 5 7
User inquiries 3 4 6
Files 7 10 15
External interfaces 5 7 10
  • FPA = indirect measure of software size (vs. SLOC = direct)
  • FPA works well for business applications but NOT for OSs, process control, engineering
  • COCOMO is more appropriate for non-business software

📌 EXAM TIP: CISA exam does NOT test FPA calculations — just understand its use and concept.

Scheduling Techniques:

Gantt Charts:

  • Show when activities begin and end on a timeline
  • Show concurrent and sequential activities
  • Show resource assignments
  • Track milestones
  • Compare progress to baseline
  • Best for: Showing schedule visually and tracking progress

Critical Path Method (CPM):

  • Critical path = Longest path through a project network
  • Activities on the critical path have zero slack time
  • Slack time = Latest possible completion time − Earliest possible completion time
  • One critical path per network (usually)
  • Used to identify the shortest possible project completion time
  • Supports resource leveling (leveling peaks and valleys)

PERT (Program Evaluation Review Technique):

  • CPM-type technique using three estimates per activity:
  • Optimistic (best case)
  • Most likely (based on experience)
  • Pessimistic (worst case)
  • PERT Formula: (Optimistic + Pessimistic + 4 × Most Likely) / 6
  • Follows a beta statistical distribution → allows confidence levels
  • Best for projects with uncertain durations (e.g., R&D, complex software)
  • PERT advantage over CPM: Incorporates probability/uncertainty

Timebox Management:

  • Fixed time period + predetermined resources
  • Used in prototyping and RAD approaches
  • Key advantage: Prevents cost overruns and delays
  • Design/development phase shortened via newer tools
  • UAT and system testing performed together

📌 Comparison: Gantt vs CPM vs PERT

Feature Gantt Chart CPM PERT
Shows timeline ✅ ❌ ❌
Shows dependencies Partially ✅ ✅
Handles uncertainty ❌ ❌ ✅
Identifies critical path ❌ ✅ ✅
Best for Visual scheduling Complex projects Uncertain projects

2.12 Project Execution (3.1.11)

  • PM + PMO start actual execution after planning is complete
  • Monitoring of internal team production and quality metrics begins
  • Key success factor: oversight of integrated team across requirements → architecture → design → development → testing → implementation → production

2.13 Project Controlling and Monitoring (3.1.12)

Management of Scope Changes:

  • Changes to scope → changes in activities → impact on deadlines and budget
  • Only stakeholders can submit change requests
  • Change requests go to → Project Manager → Change Advisory Board → Project Sponsor (for final approval/rejection)
  • All change requests must be archived

Earned Value Analysis (EVA):

Used to determine if actual spending is in line with planned spending.

EVA Metrics: - Budget to date - Actual spending to date - Estimate to complete - Estimate at completion

💡 Simple Example: If a 3-day task (8hrs/day = 24hrs total) has 8 hrs spent, but the employee says they need 20 more hours (instead of 16) → project is overrun!

Management of Risk:

Two categories of project risk: 1. Risk impacting business benefits → Project Sponsor is responsible 2. Risk impacting the project itself → Project Manager is responsible

2.14 Project Closing (3.1.13)

  • System handed over to users and/or system support staff
  • Outstanding issues must be assigned
  • Project Sponsor must be satisfied before closure
  • Relevant documentation and duties are handed off
  • Team identifies lessons learned
  • Metrics: TCO (Total Cost of Ownership) and ROI

Post-project review (formal) = done at end of project Postimplementation review = done months AFTER the project is in production

2.15 IS Auditor's Role in Project Management (3.1.14)

Tracking information in change management system includes:

  • History of all work order activity (date, programmer assigned, changes made, date closed)
  • History of logons/logoffs by programmers
  • History of program deletions
  • Adequacy of SoD and QA activities

IS Auditor Tasks Include:

  1. Meet with key development team members to identify control areas
  2. Discuss appropriate controls to rank major risks
  3. Evaluate available controls and advise on system design
  4. Periodically review documentation and deliverables
  5. Identify and test existing controls for production library security
  6. Review and analyze test plans
  7. Analyze test results and audit evidence
  8. Participate in postimplementation reviews

SECTION 3: BUSINESS CASE AND FEASIBILITY ANALYSIS (3.2)

3.1 Business Case

  • Business Case = Information required for an enterprise to decide whether a project should proceed
  • Answers: "Why should this project be undertaken and/or continued?"
  • Should be a key element of decision process throughout the project life cycle
  • Stage gates / Kill points = Decision points at which business case is formally reviewed

📌 If business case changes during an IT project → project must be reapproved through departmental planning and approval process

3.2 Feasibility Study

A feasibility study is an early study to assess if a solution is practical within established budgets and schedules.

Six Elements of a Feasibility Study:

# Element Description
1 Project Scope Clear, concise definition of the business problem/opportunity
2 Current Analysis Understanding of existing system; identifies strengths and weaknesses
3 Requirements Definition of requirements based on stakeholder needs and constraints (business, contractual, regulatory, end-user, technical)
4 Approach Course of action to satisfy requirements; identifies alternatives and rationale (build vs. buy)
5 Evaluation Examination of cost-effectiveness; includes cost comparison, schedule, ROI
6 Review Formal review by all stakeholders; approve/reject/request corrections; all sign the document if approved

🧠 Mnemonic for Feasibility Study: "SCRAE R" — Scope, Current, Requirements, Approach, Evaluation, Review

3.3 IS Auditor's Role in Business Case / Feasibility Study

When reviewing a feasibility study, the IS auditor should: - Review and evaluate the criticality of business and IS requirements - Determine if existing systems can solve the problem - Determine reasonableness of chosen solution - Determine if all cost justifications/benefits are verifiable - Review documentation for reasonableness

For requirements definition, IS auditor should: - Verify accuracy and completeness via user interviews - Identify key team members; verify all user groups were represented - Verify management approval of project initiation and cost - Review conceptual design specifications for user needs - Review UAT specifications - Determine if application is a candidate for embedded audit routine

SECTION 4: SYSTEM DEVELOPMENT METHODOLOGIES (3.3)

4.1 Business Application Development (3.3.1)

Two Types of Business Applications:

Type Objective Development Approach
Organization-Centric Collect, store, share information on need-to-know basis; regulatory compliance Traditional SDLC or software engineering
End-User-Centric Different views of data for performance optimization; includes DSS, GIS Alternative development approaches

Business Application Projects Triggered By:

  • New opportunity related to existing/new business process
  • Problem with existing business process
  • New technology opportunity
  • Problem with current technology
  • Need to align with business partners/industry standards

Key: Business objectives must be SMART — translated into key business drivers for all affected parties

4.2 SDLC Models (3.3.2)

Three SDLC Models:

1. Traditional Waterfall Model

  • Sequential, systematic approach
  • Each phase flows into the next (like a waterfall)
  • Best when: requirements are stable and well-defined
  • Primary advantage: Provides a template for all methods

Waterfall Problems: - Unanticipated events create iterations (difficulty) - Difficult to get explicit requirements upfront - User patience required (no working system until late) - Changing business environment alters requirements before delivery

2. V-Shaped Model (Verification and Validation)

  • Emphasizes relationship between development phases and testing levels
  • Most granular testing (unit test) occurs right after coding
  • Highly disciplined with strict phases and verification throughout
User Requirements ←→ Acceptance Testing Functional Req ←→ System Test Architectural ←→ Integration Test Detailed Design ←→ Unit Test [CODING]

V-Shape Problems: - Inflexible for complex or high-change projects - Concurrent dependencies may cause delays - Overreliance on documentation → less time for development

3. Iterative Model

  • Cyclical process: requirements → design → build → test → repeat
  • Each iteration incrementally improves the process
  • Suitable for large projects; allows independent features delivered periodically

Iterative Problems: - More complex project management needed - Risk analysis required more frequently by qualified resources - Overall project completion date may be ambiguous

📌 Comparison Table:

Feature Waterfall V-Shaped Iterative
Flexibility Low Low High
Testing emphasis End Throughout End of each iteration
Best for Stable requirements Small, disciplined projects Large/complex projects
Documentation High Very High Low-Medium

4.3 SDLC Phases (3.3.3)

The 6 SDLC Phases:

Phase Name Key Activity
1 Feasibility Study Determine strategic benefits; develop business case; identify ROI
2 Requirements Definition Define what system should do; identify stakeholders and requirements
3A Software Selection & Acquisition (purchased) RFP to vendors; evaluate and select vendor package
3B Design (in-house) Detailed system design; flowcharts; program specs; test plans
4A Configuration (purchased) Configure system to enterprise needs; set control parameters
4B Development (in-house) Coding; unit testing; data conversion programs
5 Final Testing & Implementation UAT; certification/accreditation; production migration
6 Postimplementation Review adequacy; measure ROI; document lessons learned

💡 Purchased software replaces Design + Development with Selection + Configuration

Phase 2 — Requirements Definition (Critical):

The IS auditor should pay close attention to: - Involvement of security engineering team - CIA (Confidentiality, Integrity, Availability) controls throughout data life cycle - Adequate audit trails defined as part of the system - Regulatory, statutory and legal requirements

Phase 3B — Design Key Activities:

  • System flowcharts and entity relationship models
  • Inputs/outputs (screen designs, reports)
  • Processing steps and computation rules
  • Data file/database design
  • Program specifications
  • Test plans for: Unit, Subsystem, Integration, Interface, Load/Init, Stress, Security, Backup/Recovery
  • Data conversion plans
  • Risk assessment of data flows (including personal data protection)

Phase 3B — Software Baselining (Design Freeze):

  • Software Baseline = Cutoff point in design; also called design freeze
  • Changes only after cost-benefit analysis and formal approval
  • Prevents scope creep (requirements changing during development)
  • Establishes configuration management (version numbers for work products)
  • Work products = functional requirements, specifications, test plans

Phase 4B — Development Key Activities:

  • Program coding standards (internal documentation, data declaration, I/O techniques)
  • Structured programming techniques (cohesion and coupling)
  • Cohesion = Each program performs a single dedicated function
  • Coupling = Independence from other programs
  • Higher cohesion + lower coupling = easier maintenance

Programming Debugging Tools:

Tool Description
Logic Path Monitors Report sequence of events performed; identify logic errors
Memory Dumps Picture of internal memory at a point in time; identify data inconsistencies
Output Analyzers Compare expected vs. actual results

Programming Languages:

Category Examples
High-level general-purpose COBOL, C
Object-oriented C++, Java
IDEs Visual Studio, JBuilder
Web scripting HTML, JavaScript, CSS
Data-centric R, Scala, SAS
Scripting Python, Perl, Shell

Risks Associated with Software Development:

Risk Type Description
Strategic Risk Business goals weighted without enterprise strategy
Business/Benefit Risk New system doesn't meet business needs/expectations
Project/Delivery Risk Exceeds financial resources; delivered late or not at all
- Within project Wrong requirements; unmanaged time/cost
- With suppliers Failure to communicate; late/deficient delivery
- Within enterprise Stakeholders not committed; changing priorities
- External environment Customers, competitors, regulations, economic conditions
- With technology Chosen tech displaced; marketplace incompatibility

Root cause of problems: Lack of discipline in managing software development process

4.4 Software Development Methods (3.3.5)

Method 1: Prototyping (Evolutionary Development)

  • Quickly builds a working model to test aspects of design and get early user feedback
  • Two approaches: 1. Build model → create design → develop real system 2. Gradually build actual production system using 4GL

Prototyping Risks: - Pressure to implement early prototype without refinement - Missing controls (backup recovery, security, audit trails) - Change control becomes complicated (changes rarely documented) - Functions/extras added beyond initial requirements

Method 2: Rapid Application Development (RAD)

Enables fast development of strategically important systems while reducing cost and maintaining quality.

RAD uses: - Small, well-trained development teams - Evolutionary prototypes - Integrated power tools (modeling, prototyping, component reusability) - Central repository - Interactive requirements and design workshops - Rigid limits on development time frames

RAD's 4 Major Stages: 1. Concept Definition — Business functions and scope 2. Functional Design — Workshops; model data/processes; working prototype 3. Development — Physical database and application; conversion system; user aids 4. Deployment — Final testing, training, data conversion, implementation

RAD supports individual application development; does NOT support planning/analysis for the enterprise as a whole

Method 3: Agile Development

Core philosophy: Not all requirements can be articulated upfront → use iterative, incremental approach

Common Agile methods: - Scrum (most popular) - Extreme Programming (XP) - Crystal, ASD, FDD, DSDM

Agile Characteristics: - Small, time-boxed subprojects (sprints in Scrum) - Replanning at end of each iteration - Heavy reliance on tacit knowledge (knowledge in people's heads) - Small co-located teams with business AND technical representatives - Daily stand-up meetings with strict time limits - Pair-wise programming (two persons code same part) - Project manager as facilitator and advocate (not planner/monitor) - Planning and control delegated to the team

Agile does NOT emphasize: - Detailed documentation - Managing a requirements baseline - Defined and repeatable processes

📌 EXAM TIP: Agile relies heavily on tacit knowledge. Documentation is secondary to working functionality.

Agile vs. Traditional SDLC Comparison:

Feature Traditional SDLC Agile
Requirements All defined upfront Evolve iteratively
Documentation Heavy Minimal
Planning Full project plan Only next iteration
Testing At end Continuous
Change Controlled Embraced
User involvement Periodic Continuous
Team size Large Small
Tacit knowledge Low reliance High reliance

Method 4: Object-Oriented System Development (OOSD)

  • Groups data and procedures into objects (contrast to traditional: data separate from procedures)
  • Object attributes = data
  • Object methods = functionality
  • Objects created from classes (templates)
  • Superclasses = parent; Subclasses = inherit parent characteristics
  • Polymorphism = same message interpreted differently by different object types
  • UML (Unified Modeling Language) = standard notation for object-oriented projects

OOSD Advantages: - Manages unrestricted variety of data types - Models complex relationships - Meets demands of changing environment

Applications: Web apps, e-business, AI, office automation, CAM

Method 5: Component-Based Development

  • Assembling applications from pre-written executable packages (components)
  • Components communicate through defined interfaces

Types of Components: | Type | Description |------|-------------| In-process client | Run within a parent container (app, VM, applet) | Stand-alone client | Well-known examples: Excel, Word | Stand-alone server | Web services, app servers, LDAP directories | In-process server | Run on servers within containers (e.g., Oracle JavaBeans) |

Benefits: Reduces time, improves quality, allows focus on business functionality, promotes modularity, reduces cost, supports multiple environments

Risks: - Lack of interoperability standards between components - Reduced system reliability - Unintended system use - Increased vulnerability/attack surface - Licensing problems

Method 6: Web-Based Application Development

  • Uses SOAP (Simple Object Access Protocol) — XML-based; defines APIs; loosely coupled components
  • WSDL (Web Services Description Language) — XML-based; identifies SOAP specification
  • UDDI (Universal Description, Discovery and Integration) — XML-based; electronic directory for web services
  • Buffer overflows remain a key risk (existed since early programming)

Method 7: Software Reengineering

  • Updates existing system by extracting and reusing design/program components
  • BPR (Business Process Reengineering) = thorough analysis and significant redesign of business processes for better performance

BPR Steps: 1. Define areas to review 2. Develop a project plan 3. Gain understanding of the process under review 4. Redesign and streamline the process 5. Implement and monitor the new process 6. Establish a continuous improvement process

⚠️ IS Auditor's BPR Concern: Key controls may be reengineered out of business processes. If a preventive control is removed, management must be made immediately aware and accept the potential risk. The IS auditor does NOT recommend reinserting all old controls.

Benchmarking Process:

6 steps: Plan → Research → Observe → Analyze → Adopt → Improve

Reverse Engineering:

  • Studying a system to see how it works → use info to develop a similar system
  • Methods: Decompiling object/executable code; black-box testing
  • Advantages: Faster development; introduces improvements
  • Risk: Software license agreements often prohibit reverse engineering!

DevOps / DevSecOps:

  • DevOps = Integration of development and operations to eliminate conflicts/barriers
  • Changes control environment → IS auditor must ensure proper SoD
  • DevSecOps = Promotes security-by-design; reduces likelihood of vulnerabilities

4.5 System Development Tools (3.3.6)

CASE (Computer-Aided Software Engineering):

Automated tools for software development process.

Category Description
Upper CASE Business and application requirements documentation (data objects, relationships, process definitions)
Middle CASE Detailed design (screen layouts, editing criteria, data organization, process flows)
Lower CASE Program code and database generation from detailed design specs

IS Auditor CASE concerns: - CASE tools help design but do NOT ensure design is correct - CASE tools must complement existing methodology - Integrity of data moved between CASE tools must be monitored - Changes to application must be reflected in CASE data - CASE repository must be secured (need-to-know) with strict version control

Fourth-Generation Languages (4GLs):

Characteristics: - Nonprocedural (event-driven, object-oriented concepts) - Environmental independence (portable across architectures) - Software facilities (screen design, graphical outputs) - Programmer workbench concepts (part of IDE) - Simple language subsets for less-skilled users

4GL Categories: | Category | Description |----------|-------------| Query/Report Generators | Extract and produce reports (audit software) | Embedded Database 4GLs | Self-contained DBMS; may not integrate well with other systems | Relational Database 4GLs | High-level products for DBMS systems (SQL+, MANTIS) | Application Generators | Generate 3GL code (COBOL, C) for further customization |

4.6 Hardware/Software Acquisition (3.3.8)

RFP (Request for Proposal) / ITT (Invitation to Tender) = specifications distributed to vendors

Acquisition Steps:

  1. Testimonials/visits with other users
  2. Provisions for competitive bidding
  3. Analysis of bids against requirements
  4. Comparison using predefined evaluation criteria
  5. Analysis of vendor's financial condition
  6. Analysis of vendor's maintenance and support capability
  7. Review of delivery schedules
  8. Pedigree of hardware (verify not from gray market)
  9. Analysis of security and control facilities
  10. Review and negotiation of price
  11. Review of contract terms (warranties, penalties, right-to-audit clauses)
  12. Prepare formal written report summarizing analysis

Vendor Evaluation Criteria:

Term Definition
Project Schedule Vendor's timetable for start, end, milestones
Turnaround Time Time to fix a problem from logging
Response Time Time system takes to respond to a query
Throughput Useful work per unit time (Kbps, Mbps, Gbps)
Workload Capacity to handle required volume
Compatibility Existing apps can run on new system
Capacity Handle simultaneous requests and data volume
Utilization System availability vs. downtime

SECTION 5: CONTROL IDENTIFICATION AND DESIGN (3.4)

5.1 Application Controls (3.4.1)

Application Controls = Controls over input, processing and output functions

Goals: - Only complete, accurate and valid data entered and updated - Processing accomplishes the correct task - Processing results meet expectations - Data are maintained

Controls achieve: data integrity and reliability → systems maintain integrity → functions operate as intended → information is relevant, reliable, secure and available

Input/Origination Controls

Input Authorization Types: | Type | Description |------|-------------| Signatures on batch forms | Evidence of proper authorization | Online access controls | Only authorized individuals can access/perform functions | Unique passwords | Enhanced by MFA; provide accountability | Terminal/workstation identification | Limits input to specific devices | Source documents | Record the data; well-designed docs increase speed and accuracy | Input data validation | Ensures expected format; prevents malicious activity |

Accountability Controls (Input): - SoD — No individual can: originate AND authorize AND verify AND distribute - Activity reports — Details of activity volume and hours; reviewed for authorized hours - Violation reports — Unsuccessful/unauthorized access attempts; indicates terminal, date, time

Batch Controls and Balancing

Types of Batch Control Totals: | Type | Description |------|-------------| Total Monetary Amount | Sum of monetary values must match batch documents | Total Items | Count of items in batch must match processed count | Total Documents | Count of documents in batch must match processed count | Hash Totals | Total of non-value numeric fields (dates, customer numbers) — has no informative value alone; provides completeness assurance |

Batch Balancing Types: - Batch Registers — Record batch totals for comparison with system totals - Control Accounts — Initial edit file determines totals; reconciled with master file - Computer Agreement — Compares batch header details to calculated totals; accept or reject

Input Error Handling Options: | Option | Description |--------|-------------| Reject only error transactions | Rest of batch processed | Reject whole batch | Batch held for correction | Hold batch in suspense | Not rejected; held pending correction | Accept and flag errors | Batch processed; error transactions flagged |

Data Validation Edits (CRITICAL EXAM AREA!)

Edit Type Description Example
Sequence Check Control numbers follow sequential order; duplicates rejected Invoices numbered 12001-15045; number >15045 is rejected
Limit Check Data should not exceed predetermined amount Payroll check ≤ $4,000
Range Check Data within predetermined range of values Product codes 100-250 only
Validity Check Data matches predetermined criteria Marital status only M or S
Reasonableness Check Data matched to predetermined reasonable limits Widget order > 20 flagged
Table Lookups Data complies with values in a computerized table City code 1-10 matched to city name table
Existence Check Data entered correctly and agrees with valid criteria Valid transaction code must exist
Key Verification Keying repeated by separate person; compared to original Worker number keyed twice and compared
Check Digit Mathematical value appended to data to detect errors DETECTS TRANSPOSITION AND TRANSCRIPTION ERRORS
Completeness Check Field should not be blank/zeros Worker number blank → record rejected
Duplicate Check New transactions matched to previously entered ones Vendor invoice not already in system
Logical Relationship Check If condition true, then other conditions must also be true Hire date > 16 years past date of birth

Output Controls = Assurance that data delivered to users is presented, formatted and delivered consistently and securely

SECTION 6: SYSTEM READINESS AND IMPLEMENTATION TESTING (3.5)

6.1 Testing Overview

Testing Purpose: Verifies and validates that a program, subsystem or application: - Performs functions for which it was designed - Operates without malfunction - Does not adversely affect other system components

QAT vs. UAT:

| QAT (Quality Assurance Testing) | UAT (User Acceptance Testing) |---|---|---| Focus | Technical aspects; documented specs | Functional aspects; user requirements | Performed by | IT department primarily | Business users | Tests | Logical design and technology | Business process scenarios | Outcome | System works as documented | System is production-ready | Combined? | NO — must NOT be combined | NO — separate objectives |

6.2 Testing Classifications (3.5.1)

Test Type Description
Unit Testing Individual program/module; tests control structure and internal operation
Interface/Integration Testing Tests connection of two or more components passing information; evaluates inter-system data flow
System Testing All modified programs tested collectively; done in non-production environment
— Recovery Testing Tests system's ability to recover after failure
— Security Testing Checks access controls; no security holes in new system
— Load Testing Large data quantities; evaluates peak performance
— Volume Testing Incremental data volume; finds maximum data capacity
— Stress Testing Incremental concurrent users/services; finds maximum concurrent user capacity
— Performance Testing Compares system performance to equivalent systems using benchmarks
Final Acceptance Testing Done in implementation phase; has two parts: QAT and UAT

Other Types of Testing:

Type Description
Alpha Testing Internal users; first stage; also called system testing
Beta Testing Limited external users; form of UAT; real-world exposure
Pilot Testing Focuses on specific/predetermined aspects; preliminary evaluation; POC is an early pilot test
White Box Testing Assesses program logic; uses test data to test logic paths (unit and integration testing) — cannot test ALL paths in large systems
Black Box Testing Tests functional operating effectiveness without regard to internal structure; applicable to integration and UAT
Function/Validation Testing Tests functionality against detailed requirements (Are we building the right product?)
Regression Testing Reruns test scenarios after changes/corrections to ensure no new errors introduced; uses same data as original
Parallel Testing Test data fed into both old and new systems; results compared; verifies new system performs same as original and meets requirements
Sociability Testing Confirms new/modified system can operate in target environment without impacting existing systems

📌 Common Exam Q: Purpose of parallel testing = ensure new system meets end-user requirements (NOT cost-effectiveness, not unit testing)

6.3 Data Integrity in Online Transaction Processing — ACID Principle

Principle Description
A — Atomicity Transaction is ALL-or-NOTHING; if error occurs, all changes backed out
C — Consistency Every transaction takes database from one consistent state to another
I — Isolation Each transaction isolated from others; accesses only consistent data
D — Durability Committed transactions survive subsequent hardware/software failures

🧠 Mnemonic: ACID — like acid, once it touches something, the effect is complete and permanent!

6.4 Testing Application Systems (3.5.4)

Technique Description Advantages Disadvantages
Snapshot Records flow of designated transactions through logic paths Verifies program logic Requires extensive IS knowledge
Mapping Identifies untested program logic; analyzes programs during execution Identifies unused code; identifies exposures Software cost
Tracing and Tagging Tracing shows trail of instructions; tagging places indicator on transactions Exact picture of event sequence Requires extensive computer time
Test Data/Deck Simulates transactions through real programs No source code review needed; objective review Risk of not including all scenarios; requires good IS knowledge
Base-Case System Uses comprehensive test data sets from evaluation Comprehensive testing; verifies operations before acceptance Extensive effort to maintain; close cooperation required
Parallel Operation Processes production data through both old and new programs; compares results Verifies new system before discontinuing old Added processing costs
Integrated Testing Facility (ITF) Creates fictitious file in database; test transactions processed with live data Periodic testing without separate test process Needs careful planning; test data must be isolated
Parallel Simulation Simulates application program logic using computer programs Eliminates need to prepare test data Programs must be developed
Transaction Selection Audit software screens and selects transactions from production Independent of production system Development and maintenance cost
Embedded Audit Data Collection Software embedded in host application; selects and generates transactions Sampling and production statistics High development cost; auditor independence issues
— SCARF Auditor determines reasonableness of tests Info for further review
— SARF Randomly selects representative transactions Analysis of representative file
Extended Records Gathers all data affected by a particular program Records in one convenient file Adds data storage costs

6.5 IS Auditor's Role in Testing

  1. Review test plan for completeness and user participation evidence
  2. Reconcile control totals and converted data
  3. Review error reports for precision in recognizing erroneous data
  4. Verify cyclical processing correctness
  5. Verify accuracy of critical reports
  6. Interview end users about understanding of new procedures
  7. Review system documentation for completeness
  8. Verify system security functioning (develop and execute access tests)
  9. Review unit and system test plans for internal control testing
  10. Ensure accepted software delivered to implementation team (vendor cannot replace this version)

6.6 System Implementation (3.5.5)

  • Implementation only after successful testing
  • Must follow enterprise change control procedures
  • IS auditor verifies appropriate signoffs obtained before implementation
  • Verify all data conversions are correct and complete before production implementation

SECTION 7: IMPLEMENTATION CONFIGURATION AND RELEASE MANAGEMENT (3.6)

7.1 Overview

Configuration Management = Systematic, consistent control on attributes of IT components (hardware, software, firmware, network connectivity)

Key tools: ManageEngine Service Desk, SysAid, ServiceNow

Changes to IT systems must be: Carefully assessed → Planned → Tested → Approved → Documented → Communicated

7.2 Configuration Management Systems (3.6.1)

  • Maintenance requests must be formally documented and approved by a change control group (Configuration Control Board)
  • Uses CMDB (Configuration Management Database) — documents hardware and software assets

Check-In/Check-Out Process: - Check-out = Prevents/manages simultaneous code edits; requires approved change request - Check-in = Moving changed item to controlled environment with new version number

Configuration Management Plan should include: - All system documentation - Test plans and procedures - Software developed

Configuration Management Tools Support: 1. Identifying items affected by proposed change (impact assessment) 2. Recording configuration items affected by authorized changes 3. Implementing changes per authorization records 4. Registering configuration item changes 5. Recording baselines for recovery (rollback) 6. Preparing releases to avoid human errors

CIS Benchmarks = Configuration baselines from Center for Internet Security for securing systems

SECTION 8: SYSTEM MIGRATION, INFRASTRUCTURE DEPLOYMENT AND DATA CONVERSION (3.7)

8.1 Data Migration (3.7.1)

Data Conversion (Data Porting) = Required when source and target systems use different field formats, sizes, database structures or coding schemes

Data Conversion Objective: Convert existing data to new format while preserving meaning and integrity

Steps for Successful Data Conversion: 1. Determine which data to convert via programs vs. manually 2. Perform data cleansing before conversion 3. Identify verification methods (file comparisons, record counts, control totals, sampling) 4. Establish parameters for successful conversion 5. Schedule conversion tasks 6. Design audit trail reports (data mappings and transformations) 7. Design exception reports 8. Establish responsibility for sign-off 9. Develop and test conversion programs 10. Perform conversion dress rehearsals 11. Control outsourcing (nondisclosure, data privacy, data destruction) 12. Run actual conversion with all necessary personnel

IS Auditor's Role in Data Migration: - Ensure tools and techniques selected are adequate - Data conversion achieves objectives without data loss or corruption - Any data loss is minimal and formally accepted by user management

8.2 Changeover (Go-Live / Cutover) Techniques (3.7.2)

Three Changeover Techniques:

Technique Description Risk Level Key Characteristics
Parallel Old and new systems run simultaneously; results compared; user gains confidence before old discontinued Lowest Most costly; minimizes risk; old system discontinued after confidence gained
Phased Old system broken into modules; new system modules replace old ones sequentially Medium Step-by-step; module by module replacement
Abrupt (Cold Turkey/Direct Cutover) Old system discontinued at cutoff date/time; new system takes over immediately Highest Fastest; least costly; highest risk

Phased Changeover Risks: - Resource challenges (maintaining two unique IT environments) - Extension of project life cycle to cover two systems - Change management for requirements and customizations

Abrupt Changeover Steps: 1. Conversion of files and programs; test running on test bed 2. Installation of new hardware, OS, application and migrated data 3. Training employees in groups 4. Scheduling operations and test running for go-live

Risk items related to changeover: - Asset safeguarding - Data integrity - System effectiveness and efficiency - Change management challenges - Duplicate or missing records

8.3 System Change Procedures and Program Migration Process (3.7.3)

IS auditor should review: - Methodology for authorizing, prioritizing and tracking system change requests - Whether emergency change procedures are in operations manuals - Whether change control is a formal procedure - Whether change control log shows all changes resolved - Adequacy of security access restrictions over production source and executable modules

8.4 Certification/Accreditation (3.7.5)

Term Definition Who does it
Certification Comprehensive assessment against a standard (management, operational, technical controls); determines extent to which controls are correctly implemented Assessor enterprise
Accreditation Official management decision by senior official to authorize operation; accepts risk to enterprise operations, assets or individuals Senior official
  • Accreditation = Official authorization to operate the system
  • Senior official who accredits accepts full responsibility and accountability for security of the system
  • If breach occurs, the accrediting official is held accountable

SECTION 9: POSTIMPLEMENTATION REVIEW (3.8)

9.1 What is Postimplementation Review?

  • Postimplementation Review = Carried out weeks or months after project completion (when benefits and shortcomings are realized)
  • NOT the same as project closure (project closure = at end of project)
  • Part of benefits realization process
  • Used to determine whether appropriate controls were built into the system

9.2 What Does Postimplementation Review Consider?

  1. Adequacy of the system: - Does system meet user requirements and business objectives? - Have controls been adequately defined and implemented?

  2. Projected cost vs. benefits / ROI measurements

  3. Recommendations for system inadequacies

  4. Plan for implementing recommendations

  5. Assessment of development project process: - Were chosen methodologies, standards and techniques followed? - Were appropriate project management techniques used? - Is risk of operating the system within acceptable levels?

9.3 Project Closeout Steps (Figure 3.28):

Step Action
1 Assign responsibility for outstanding issues; identify related budget
2 Assign custody of contracts; archive or transfer documentation
3 Conduct postimplementation review with project team, developers, users, stakeholders; identify lessons learned
4 Document identified risks; update risk register
5 Complete second postimplementation review after deliverables have been in production long enough to realize true business benefits and costs

9.4 Measurements of Critical Success Factors (Figure 3.29):

Category Measures
Productivity Dollars spent per user; transactions per month; transactions per user
Quality Number of discrepancies; number of disputes; fraud/misuse detection
Economic Value Total processing time reduction; monetary value of admin costs
Customer Service Turnaround time for customer questions; frequency of useful communications

9.5 IS Auditor's Role in Postimplementation Review (3.8.1)

Key Rule: IS auditor performing postimplementation review must be independent of the system development process. An IS auditor who consulted on the system cannot perform this review.

IS Auditor Tasks: 1. Determine if system objectives and requirements were achieved (end user use, trouble tickets, work orders, satisfaction) 2. Determine if cost benefits from feasibility study are being measured and reported 3. Review program change requests (type of changes indicate design/programming problems) 4. Review controls built into system to ensure operating per design 5. Review operator error logs for resource/operating problems 6. Review input and output control balances and reports

SECTION 11: PROBABLE EXAM QUESTIONS

High-Probability Questions with Answers

Q1. A project manager in a functional organization structure has:

A. No formal management authority — can only advise B. Full authority over budget and team C. Shared authority with department heads D. Authority over schedule only

Answer: A — In functional structure, PM can only advise

Q2. Which data validation edit is MOST effective in detecting transposition and transcription errors?

A. Range check B. Limit check C. Check digit D. Validity check

Answer: C — Check digit is mathematically calculated to detect altered data, transpositions (123→132) and transcriptions (123→124)

Q3. The PRIMARY purpose of parallel testing is to:

A. Determine whether the system is cost-effective B. Enable comprehensive unit and system testing C. Highlight errors in program interfaces with files D. Ensure the new system meets user requirements

Answer: D — Parallel testing compares results of old and new systems to confirm new system meets end-user requirements

Q4. When introducing thin client architecture, which type of risk is significantly increased regarding servers?

A. Integrity B. Concurrency C. Confidentiality D. Availability

Answer: D — Thin client makes servers critical to operation; server failure probability increases = increased availability risk

Q5. In agile development methodology, which of the following is MOST commonly associated?

A. Reliance on detailed documentation B. Reliance on strict standard operating procedures C. Lack of user requirements D. Heavy reliance on tacit knowledge

Answer: D — Agile teams rely on knowledge in people's heads (tacit knowledge) rather than documentation

Q6. When a software package is purchased instead of developed in-house, which SDLC phases are replaced?

A. Design and development phases → replaced with selection and configuration phases B. Feasibility and requirements phases C. Implementation and testing phases D. Nothing is replaced

Answer: A — Selection and configuration replace design and development for purchased packages

Q7. To assist in testing a banking system with sensitive production data, the IS auditor's PRIMARY concern is that the test data should be:

A. Sanitized B. Complete C. Representative D. Current

Answer: A — Test data MUST be sanitized to prevent sensitive data leaking to unauthorized persons

Q8. User specifications in a waterfall SDLC project have not been met. The MOST likely source of this issue is:

A. Quality assurance B. Requirements C. Development D. User training

Answer: B — If user specifications are not met, requirements engineering failed to correctly define/document what users needed

Q9. During BPR, an IS auditor finds an important preventive control has been removed. The auditor should:

A. Inform management immediately and determine if they're willing to accept the potential material risk B. Determine if a detective control replaced it; if not, report removal C. Recommend all old controls be reinstated D. Develop continuous audit approach

Answer: A — Management must be immediately informed and given the choice to accept the risk. Other actions come AFTER management decision.

Q10. Which weakness is MOST significant in ERP software used by a financial enterprise?

A. Access controls have not been reviewed B. Limited documentation C. Two-year-old backup media not replaced D. Database backups once a day

Answer: A — In a financial enterprise, unreviewed access controls have the most serious consequences given sensitive data and assets accessible

Q11. PERT calculates activity duration estimates using:

A. One estimate per activity B. Two estimates (optimistic and pessimistic) C. Three estimates: optimistic, most likely, pessimistic D. Four estimates based on historical data

Answer: C — PERT formula: (O + P + 4×ML) / 6

Q12. A software escrow agreement ensures:

A. Software remains free of bugs B. Vendor performs regular updates C. Source code is held by a third party and released if vendor goes out of business D. Enterprise retains full ownership of software

Answer: C — Escrow protects enterprise by providing access to source code if vendor ceases operations

Q13. The ACID principle's "Atomicity" means:

A. Transactions are isolated from each other B. Database maintains consistency between transactions C. A transaction is all-or-nothing; if error occurs, all changes are backed out D. Committed transactions survive hardware failures

Answer: C — Atomicity = complete or nothing; partial commits are not allowed

Q14. Which changeover technique carries the LOWEST risk?

A. Parallel changeover B. Phased changeover C. Abrupt changeover D. All carry equal risk

Answer: A — Parallel allows both old and new systems to run simultaneously until confidence is gained

Q15. Certification differs from accreditation in that:

A. Certification is performed by senior management; accreditation by auditors B. Certification assesses compliance with controls; accreditation is the official authorization decision to operate C. Accreditation is more technical than certification D. They are identical processes

Answer: B — Certification = comprehensive technical assessment; Accreditation = management decision to authorize operation and accept risk

Q16. A hash total is best described as:

A. The total monetary value of all transactions in a batch B. The total count of all records processed C. The total of non-value numeric fields (e.g., customer numbers, dates) used for completeness verification D. A cryptographic hash of the database

Answer: C — Hash totals aggregate non-meaningful numeric fields to verify batch completeness

Q17. An IS auditor performing a postimplementation review should:

A. Have been part of the original development team B. Report directly to the project manager C. Be independent of the system development process D. Focus primarily on technical performance metrics

Answer: C — Independence is critical; if the auditor was involved in development, they cannot objectively review the same system

Q18. The PERT formula for calculating activity duration is:

A. (Optimistic + Pessimistic) / 2 B. (Optimistic + Most Likely + Pessimistic) / 3 C. (Optimistic + Pessimistic + 4 × Most Likely) / 6 D. (2 × Most Likely + Pessimistic) / 3

Answer: C — Standard PERT formula using beta distribution weighting

12.3 IS Auditor Key Principles (Master These!)

  1. Advisory ≠ Audit: An IS auditor in an advisory role is NOT performing an audit
  2. Independence is paramount: Cannot audit what you helped build
  3. Controls first: IS auditor focuses on ensuring controls are designed, implemented, and functioning
  4. Management decides: IS auditor recommends — management makes risk acceptance decisions
  5. SoD must be protected: Even in small organizations or DevOps environments
  6. Test data must be sanitized: Never use live production data without sanitizing
  7. Postimplementation review = months after; measures actual ROI vs. planned
  8. Business case drives governance: All decisions should align with the business case

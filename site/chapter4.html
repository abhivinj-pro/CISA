PART A: INFORMATION SYSTEMS OPERATIONS

4.1 IT Components

Overview

IT operations encompass everything needed to keep an organization's systems running reliably and securely. The IS auditor must understand these components to assess risks and controls effectively.

Key IT Operation Elements:

Element What It Is Auditor Focus
Infrastructure Hardware, data centers, networking Effectiveness, efficiency, security
Network Routers, switches, firewalls, VPNs Security controls, access controls
Applications & Software OS, apps, databases, middleware Change management, access reviews
System Monitoring Log analysis, event management Performance issues detection
Backup & Disaster Recovery DRP, replication mechanisms Business continuity readiness
Virtualization & Cloud IaaS, SaaS, PaaS SOC reports, third-party reviews
Documentation Procedures, troubleshooting guides Completeness, currency

4.1.1 Networking

The OSI Reference Model

The Open Systems Interconnection (OSI) model is a 7-layer conceptual framework for network communication. Each layer is self-contained and communicates with the same layer on the remote system (peer-to-peer).

Memory Aid: "All People Seem To Need Data Processing" (Layer 7 → 1)

Layer # Layer Name Function Data Unit Key Protocols/Devices
7 Application User interface to network; provides services to apps Data HTTP, FTP, SMTP, TFTP, DNS, DHCP
6 Presentation Data translation, encryption, compression, reformatting Data SSL/TLS, encryption
5 Session Establishes, manages and terminates sessions Data NSP, SNMP, Telnet, NetBIOS
4 Transport End-to-end reliable/unreliable delivery, error recovery, flow control Segment TCP, UDP
3 Network Logical addressing (IP), routing and forwarding Packet IP, ICMP, ARP, RARP; Routers
2 Data Link Reliable transfer across physical link, MAC addressing, CRC error detection Frame Ethernet, PPP; Switches, Bridges
1 Physical Bit transmission as electrical/optical/radio signals Bits Cables, connectors, NICs, Hubs

Exam Point: Data travels DOWN the OSI stack on the sender side (headers/trailers added) and UP the stack on the receiver side (headers/trailers removed). Each layer adds control information as an "envelope."

Important OSI Layer Details: - Data Link Layer uses MAC (Media Access Control) addresses — unique hardware addresses assigned at manufacture - CRC (Cyclic Redundancy Check) detects single-bit and most multi-bit errors at Data Link layer - Promiscuous mode — a NIC configured to listen to ALL data (not just its own MAC address) — used in network monitoring/sniffing - Network Layer understands IP addresses and handles routing

TCP/IP vs OSI Model

TCP/IP was developed BEFORE the OSI model — there is no direct match between the two.

OSI Layers TCP/IP Layer Key Protocols
Application + Presentation + Session Application HTTP, FTP, SMTP, DNS, DHCP, SNMP, Telnet
Transport Transport TCP (reliable), UDP (unreliable)
Network Internet IP, ICMP, ARP, RARP
Data Link + Physical Network Interface Ethernet, PPP, Frame Relay

Local Area Network (LAN)

Definition: A network covering a small local area — a room to a few buildings.

LAN Physical Media:

Media Use Advantages Disadvantages
Twisted-pair (copper) Short distances (<200 ft), voice and data Cheap, simple, easy to install Easy to tap, cross talk, EMI interference
Coaxial cable Data and video Straightforward, readily available Thick, expensive, distance sensitive
Fiber optic Long distances, voice/data/video High bandwidth, secure, no cross talk, no EMI Expensive, hard to splice/modify, fragile
Radio (wireless) Short distances Cheap, convenient Easy to tap, interference, noise
Microwave radio Line-of-sight, voice and data Available Easy to tap, weather-sensitive
Satellite radio Wide area coverage High bandwidth Interference, delay (~300ms traditional, <100ms LEO)

LAN Topologies: - Bus — All devices on a single cable (legacy, rarely used) - Ring — Devices form a circle (legacy) - Star — All devices connect to a central switch/hub (current standard)

LAN Components:

Device OSI Layer Function
Repeater Layer 1 (Physical) Amplifies/regenerates signals; extends range
Hub Layer 1 (Physical) Multi-port repeater; broadcasts to all ports
Bridge Layer 2 (Data Link) Connects LAN segments; filters by MAC address; software-based
Layer 2 Switch Layer 2 (Data Link) Store-and-forward; reduces collision domains; hardware (ASIC); faster than bridge
Router Layer 3 (Network) Routes packets using IP addresses; logically separates networks; software-based

Exam Point: Switches are hardware-based (ASIC) and faster than software-based bridges. Routers examine every packet — can create bottlenecks.

LAN Risks: - Unauthorized data changes (integrity loss) - Version control failures - Virus/worm infection - Data disclosure (no need-to-know enforcement) - Impersonation (masquerading as legitimate user) - Sniffing — harvesting network addresses to launch attacks - Spoofing — reconfiguring address to pretend to be different - Lack of audit trails; destruction of log data

LAN Security Controls: - Principle of Least Privilege (POLP) - Record/file locking to prevent simultaneous updates - Strong password policies - Port security using switches (prevent unknown MAC addresses) - Encrypting local traffic with IPSec - Auditor documents: topology, admin/owner, user groups, applications, procedures

Wide Area Network (WAN)

Definition: Transmits data across geographically dispersed LANs — cities, nations.

Key WAN Characteristics: - Operates at Physical and Data Link layers - Data flow: Simplex (one-way), Half-duplex (one-way-at-a-time), Full-duplex (both-ways simultaneously) - Lines: Switched or Dedicated

WAN Message Techniques: - Message switching — Complete message sent to concentration point; stored and forwarded - Virtual circuits — Logical circuit between two devices: - SVC (Switched Virtual Circuit) — On-demand connectivity - PVC (Permanent Virtual Circuit) — Always-on connection

ISO WAN Management Tasks (ISO/IEC 10040): 1. Fault management — Detects technical faults 2. Configuration management — Remotely define/change device configs 3. Accounting management — Records resource usage 4. Performance management — Monitors usage, sets thresholds 5. Security management — Detects suspicious traffic, generates alarms

Network Management Tools: - Response time reports — Identify slow system response - Downtime reports — Track line/circuit availability - Online monitors — Echo checking, status checking - Network (protocol) analyzers — Hardware-based tools at data link/network level; monitor packets - SNMP (Simple Network Management Protocol) — TCP/IP-based; monitors/controls network variables; polls devices via agents; use SNMPv2 or v3 for encryption - ICMP (Internet Control Message Protocol) — Error reporting protocol; used by ping

Converged Protocols

Protocols that merge proprietary and standard protocols over existing TCP/IP infrastructure:

Protocol Description Key Point
FC (Fiber Channel) High-speed storage (up to 16 Gbps, 10 km); dedicated infrastructure More secure than IP-based storage
FCoE (Fiber Channel over Ethernet) FC frames over Ethernet; requires 10+ Gbps Expensive to implement
MPLS (Multi-Protocol Label Switching) Routes by short labels (20-bit) instead of full IP addresses; faster Protocol-independent; avoids routing table lookups
VoIP (Voice over IP) Voice/video over TCP/IP; cheaper than landline Can support videoconferencing
iSCSI SAN protocol over TCP/IP; low-cost FC alternative Easy to configure; supports DR; reusable

iSCSI Components: - Initiator — Host software/hardware that sends data to storage - Target — Storage device that receives the data

IPv4 vs IPv6

Feature IPv4 IPv6
Address size 32-bit (4.3 billion addresses) 128-bit (near-infinite addresses)
Checksum Used Not available
Transmission method Broadcasting Multicasting
Encryption/Authentication Not supported Provided
Address configuration Manual + DHCP Manual, DHCP, autoconfiguration, renumbering
End-to-end integrity Cannot be achieved Highly achievable
Security Not security-focused Enhanced for security
Packet flow identification No mechanism Flow label field in header

IPv4 → IPv6 Transition Strategies: 1. Dual stacking — Run both versions on same device 2. Tunneling — Use IPv4 routing to carry IPv6 traffic

Network Address Translation (NAT)

Definition: Converts internal IP addresses to public IP addresses for Internet communication.

Types of NAT:

Type Description
Static NAT One-to-one permanent mapping of internal to external IP
Dynamic NAT Internal IP dynamically mapped from a pool of external IPs
NPAT (NAT with Port Address Translation) Multiple devices share ONE public IP — different port numbers used

NAT Benefits: Cost reduction, improved privacy, increased security (blocks uninitiated inbound connections), flexibility, boundary enforcement

NAT Limitations: Time-consuming implementation, large storage for connection tables, may interfere with IPSec, compatibility issues

4.1.4 USB Mass Storage Devices

Risks of USB Devices: - Virus/malware vectors (very hard to defend) - Data theft (physical access opportunity) - Data/media loss (portability risk) - Data corruption (improper removal) - Compatibility issues - Physical damage - Loss of confidentiality (HIPAA violations if patient data lost)

USB Security Controls: - Encryption (renders data useless without key) - Granular control software (block unauthorized USB; allow specific manufacturers) - Personnel education - Lock desktop policy (auto-lock after short interval) - Antivirus scanning - Use only encrypted/approved devices - Include return info text file on device

Wireless Advantages: Convenience, mobility, scalability

4.3 Job Scheduling and Production Process Automation

Definition: A job schedule lists all jobs that must run, the order, and dependencies.

Why It Matters: Automated job scheduling controls the complex process of running thousands of daily file transfers.

Job Scheduling Software Advantages: - Reduces manual task burden - Job information set up only once (less operator error) - Dependencies defined (dependent jobs halt if predecessor fails) - Security over production data - Logs of all successes and failures

Scheduling Principles: - High-priority jobs → optimal resources - Maintenance (backup, reorganization) → non-peak times - Low-priority jobs → scheduled when resources available

Auditor Review Focus (Figure 4.8): - Are critical applications identified with highest priority? - Are schedules followed and rush jobs justified? - Does daily schedule serve as audit trail? - Are all job failures flagged, investigated, and communicated? - Are restarted jobs authorized and logged? - Are personnel authorized to change schedules?

4.4 System Interfaces

Definition: Allow data output from one application to be input for another with minimal human interaction.

Three Categories of Data Transfer:

Category Description Risk
System-to-system Data between two systems (internal/external); common in ERP, analytics Pervasive errors if not corrected quickly
Partner-to-partner Two+ organizations interface directly Exposes confidential data to third parties
Person-to-person Human data exchange (email attachments, shared drives) Human error, security exposure, hardest to monitor

Controls for System Interfaces: - Managed File Transfer (MFT) system for tracking - Manage multiple transfer mechanisms and APIs - Encryption/decryption of data in transit - Reconciliation (manual or automated via cryptographic hash checksums) - Non-repudiation controls (ensure intended recipient is actual recipient) - Audit trails capturing: who sent, when, when received, data structure, how sent, who received

Key Risk: Data must be the same on sending and receiving system; unauthorized access during transit is a major concern.

4.5 End-User Computing and Shadow IT

End-User Computing (EUC)

Definition: Technologies enabling non-IT professionals to create and manage technical resources.

EUC Risks (from lack of IT oversight): - No authorization mechanism - No authentication mechanism - No audit logging (Excel/Access don't have native logs) - No encryption for sensitive data - No data backup - Compliance risk (sensitive/private data) - Not in IT inventories (monitoring challenges) - Not in maintenance/patching schedules

Auditor Response: Ensure EUC policies exist; data classification applied; inventory maintained; critical EUC apps get same controls as enterprise apps.

Shadow IT

Definition: Use of systems/services/hardware/software without IT/cybersecurity department approval.

Why It Grows: Low-cost cloud software, smartphone proliferation, work-from-home, personal hardware.

Shadow IT Controls:

Control Description
Shadow IT policy Aligned with business objectives and security requirements
IT as service-delivery org Consultative IT culture encouraging business partnership
IT budgeting/procurement IT reviews all IT purchases
System consolidation Limit service providers, networks, platforms
User access/admin rights Explicitly assigned; unapproved installs prevented
User education Formal IT education program
User activity monitoring Record and monitor user behavior
User data exchange Strong end-point controls

Key Concept: Shadow IT can drive innovation but exposes enterprise to significant risk. Controls must be continuously assessed, not one-time.

4.6 Systems Availability and Capacity Management

4.6.1 IS Architecture

Computer Architecture Hierarchy (bottom to top): 1. Hardware + firmware (base) 2. Kernel/Nucleus — Handles interruptions, process creation/destruction, memory allocation, I/O support; highly privileged; most user access restricted 3. OS processes (system software) 4. Applications

System Software Examples: - Access control software - Data communications software - Database management software - Program library management - Job scheduling software - Utility programs - Network management software

4.6.2 Operating Systems (OS)

OS Functions: Interface between user, processor, and application software; manages processors, RAM, disk storage, and I/O devices.

Key OS Concept: Virtual storage — programs reference addresses beyond real memory; OS maps large slower memory to faster smaller working memory.

OS Types by Environment: - Single-user micro — Simple file cataloging + program loading - Mainframe (z/OS) — Extensive concurrent operations, batch + real-time, high reliability - Server OS (UNIX/Linux) — Multiprocessing, multitasking, multi-user, network management - Virtual environment — One physical server partitioned into multiple independent virtual servers, each running its own OS

Software Control Parameters deal with: - Data management - Resource management - Job management - Priority setting

OS Integrity Requirements: - Protect from deliberate/unintentional modification - Ensure user programs cannot interfere with privileged programs - Process isolation — prevent concurrent processes from interfering - Least privilege — Processes have no more privilege than needed

Windows Registry — Critical system configuration files; changes tracked by IS auditor for integrity, confidentiality, availability.

4.6.3 Access Control Software

Prevents unauthorized access to data, system functions, programs; prevents unauthorized updates.

4.6.4 Data Communications Software

Transmits messages/data between points. Components: Transmitter → Transmission path → Receiver

4.6.5 Utility Programs

System software for maintenance. Five functional categories: 1. Understanding application systems 2. Assessing/testing data quality 3. Testing program integrity 4. Faster program development 5. Improving operational efficiency

Security Note: Many utilities operate outside the security system without audit trails. Access to sensitive utilities must be controlled and restricted.

4.6.6 Software Licensing

Free Software Types:

Type Description
Open source May use, copy, modify, redistribute; source code included (e.g., Linux)
Freeware Free; source code cannot be redistributed (e.g., Adobe Reader)
Trial/Demo Free initially but limited functionality or time

Paid Software Types:

Type Basis
Per CPU Number of CPUs/cores
Per seat Number of unique users
Concurrent users Total users in defined period
Utilization CPU activity or active users at a time
Per device Number of devices (not users)
Enterprise Unlimited use throughout organization
Mixed Combination of types

Auditor Actions for License Compliance: 1. Review listing of all software (standard, used, licensed) 2. Obtain copies of all software contracts 3. Scan network to list installed software 4. Review server specs (CPUs, cores) 5. Compare license agreements with installed software

Prevention of License Violations: - Good software asset management - Centralized control/distribution - Disable user ability to install software - Regular endpoint scanning - Documented policies requiring user sign-off

4.6.7 Source Code Management

Source code = Human-readable programming language text; must be compiled to machine language.

Version Control System (VCS)/Revision Control Software (RCS): - Maintains central repository - Programmers "check out" → modify → "check in" (creates new revision) - Supports branching (copies of trunk for different customers/regions) - Examples: Git (distributed — DVCS), Apache Subversion (centralized)

VCS Advantages: Access control, change tracking, concurrent development, rollback, branching

Auditor Focus on Source Code: - Who has access - Who can commit (push to production) - Alignment with change/release management - Backups including offsite + escrow agreements

4.6.8 Capacity Management

Definition: Planning and monitoring computing/network resources to ensure efficient and effective use.

Key Information for Capacity Planning: - CPU/vCPU utilization - Computer storage utilization - LAN/WAN bandwidth utilization - I/O channel utilization - Number of users - New technologies/applications - SLAs

Capacity Planning and Monitoring Cycle:

Developing → Monitoring → Analyzing → Tuning → Implementing → Modeling → Application Sizing
Activity Description
Developing Create capacity plan for current/future IT resources
Monitoring Monitor IT components against agreed service levels
Analyzing Identify trends, baselines from monitoring data
Tuning Optimize systems for actual/expected workload
Implementing Introduce changes/new capacity
Modeling Forecast future capacity trends
Application sizing Predict resource needs for new capacity

Key Threshold: File server storage at 80% → install additional disk; mainframe utilization target 85-95%.

Application Stacking: Consolidating apps on fewer large servers — better resource use but higher impact if server goes down.

Capacity Monitoring is Pro-active — prevents performance problems by detecting threshold breaches before they affect business operations.

4.7 Problem and Incident Management

4.7.1 Problem Management

Definition: Resolves issues through in-depth root cause analysis of a major incident or several similar incidents.

Root Cause Analysis Methods: - Fishbone/Ishikawa diagrams — Cause-and-effect analysis - Brainstorming - 5 Whys — Iterative questioning to find root cause

Process: 1. Identify problem 2. Root cause analysis → known error 3. Develop workaround 4. Add to KEDB (Known Error Database) 5. Proactively prevent recurrence

Problem vs Incident Management:

Aspect Problem Management Incident Management
Objective Reduce number/severity of incidents Return to normal service ASAP
Approach Proactive; root cause focus Reactive; speed focus
Output Known error database, permanent fix Service restoration

4.7.2 Incident Handling Process

Incident Life Cycle: 1. Initiation/Detection 2. Classification (impact + urgency) 3. Assignment to specialists 4. Resolution 5. Closure

Priority Determination: Based on impact AND urgency (e.g., CIO's printer request vs. server crash → server crash gets priority by policy).

Escalation: Unresolved incidents escalated per IS management criteria.

4.7.3 Error Logging

Types of Errors Logged: - Application errors, System errors, Operator errors - Network errors, Telecommunication errors, Hardware errors

Error Log Entry Contents: - Error date, code, description, source - Department/position responsible - Escalation date/time - Resolution narrative and status code - Initials of individual maintaining and closing the log

Control Point: Ability to ADD to error log should not be restricted, but UPDATES should be restricted to authorized individuals. Closing an entry should be assigned to a DIFFERENT person than the one who initiated it (Separation of Duties).

4.7.4 Help Desk

Primary Purpose: First, single, and central point of contact for users.

Triage Process: Help desk ticket initiated → classified → escalated based on complexity.

Help Desk Functions: - Determine source of incidents; take corrective action - Initiate problem reports - Provide second and third-tier support - Maintain vendor software documentation - Communicate abnormal patterns to IS operations

4.8 IT Change, Configuration and Patch Management

Change Management

Purpose: Control movement of application changes from test → QA → production environments.

Change Management Covers: - Hardware modifications - Software installation/upgrades - Application patching - Network device configuration (firewalls, routers, switches)

Change Categories: - Emergency — Immediate implementation required - Major — Significant changes with full testing - Minor — Small enhancements/fixes

Change Management Ensures: - All relevant personnel informed - Documentation complete and compliant - Testing reviewed and approved by user/project management - Data file conversion accurate - Rollback plan exists for high-risk changes - Legal/compliance aspects considered

4.8.1 Patch Management

Definition: Acquiring, testing, and installing code changes to maintain up-to-date software and address security risks.

Patch Management Tasks: 1. Maintain current knowledge of available patches 2. Decide which patches apply to which systems 3. Test patches on non-critical systems first 4. Document all associated procedures 5. Backup before patching

Risk: Patches can be ineffective or introduce new problems. Always test before production deployment.

4.8.2 Release Management

Definition: Process of making software available to users; a "release" is a collection of authorized changes.

Types of Releases:

Type Description Testing Level
Major release Significant new functionality; supersedes all minor releases Comprehensive testing
Minor release Small enhancements/fixes; fixes that can't wait for major release Full release process but less demanding
Emergency release Critical fix required immediately; avoid when possible Limited testing — increases error risk

Delta release — Only includes items changed since last release (partial release).

Release Planning Includes: - Contents consensus - Release strategy (phasing by time, geography, business unit) - High-level schedule - Resource planning (including overtime) - Roles and responsibilities - Rollback plans - Quality plan

4.8.3 IS Operations

IS Operations Staff Tasks: - Execute and monitor scheduled jobs - Facilitate timely backup - Monitor unauthorized access to sensitive data - Monitor adherence to IS operations procedures - Participate in DRP tests - Monitor performance, capacity, availability, and failure of resources - Facilitate troubleshooting and incident handling

IS Operations Documentation Includes: - Operations procedures (job flows, equipment instructions) - System/application monitoring procedures - Error/problem detection procedures - Problem escalation procedures - Backup and recovery procedures

4.9 Operational Log Management

Definition

Operational logs (audit trails) track and log activities — providing historical records of actions, events, and interactions.

Key Use: Log monitoring used as a compensating control when primary controls (like separation of duties) are not feasible in small organizations.

4.9.1 Types of Logs

Log Type Purpose
Event log Network traffic, login attempts, failed passwords, application events
Server log Activities related to a specific server
System log OS events: startup, shutdowns, errors, warnings
Access log Who/what accessed applications or files; compensating control
Change log Chronological list of changes to applications/files
Availability log System uptime/performance; used for SLA compliance
Resource log Connectivity issues, capacity limits; early breach indicator
Threat log Traffic matching predefined security profile; firewall alerts
Database log Insertions, updates, deletions; maintains data integrity
Error log System problems with details for resolution
Firewall log Incoming/outgoing traffic through firewall
OS log Data file versions, programs scheduled, utility usage; verifies OS integrity
Application log Efficiency, authorization of changes, unauthorized access
Access control log Access control effectiveness for critical data, DBMSs, apps

4.9.2 Log Management Cycle

Generate → Alert → Store/Protect → Analyze → Report → Archive/Retain

Three-Tier Log Infrastructure: 1. Log generation tier — Hosts generating log data (clients/servers) 2. Log analysis and storage tier — Log servers receiving, storing, aggregating (collectors/aggregators) 3. Log monitoring tier — Consoles for monitoring, reviewing, reporting

SIEM (Security Information and Event Management): - Integrates data from multiple log sources - Real-time monitoring, event correlation, threat detection - More comprehensive than standalone log management

Log Analysis Techniques: - Parsing — Extract relevant info; normalize different formats - Event correlation — Identify relationships between events (SIEM provides this) - Pattern matching — Search for predefined attack signatures - Statistical analysis — Mean, median, std deviation, time-series analysis for anomaly detection

Log Security: - Protect from alteration (intruders try to hide by altering logs) - Centralize on secure SIEM server - Use encrypted file systems/databases - Role-based access controls - Regular automated backups (offsite/cloud) - Retention policy per regulatory requirements

At minimum, logs must capture: Timestamp, user information, event information

4.10 IT Service Level Management

Overview

ITSM manages IT through discrete processes to deliver service to the business. SLAs maintain and improve customer satisfaction.

Frameworks: - ITIL — Detailed framework for operational service management based on business value - ISO 20000 — International ITSM standard; ensures ITSM processes align with business needs

4.10.1 Service Level Agreements (SLAs)

Definition: Agreement between IT organization (internal or external) and customer detailing services provided.

SLA Characteristics: Accuracy, completeness, timeliness, security

Monitoring Tools for SLA Compliance:

Tool Purpose
Exception reports Identify applications that didn't complete successfully
System and application logs Identify application problems
Operator problem reports Manual logs of computer operations problems
Operator work schedules Ensure proper staffing for service requirements

When SLA terms not met: Service provider may pay fines or reduce billing per contractual obligations.

OLA (Operational Level Agreement): Internal agreements covering delivery of services that support IT's delivery to business.

4.10.2 Monitoring Service Levels

  • Must be regularly monitored by appropriate management level
  • Especially critical for outsourced services
  • Key principle: When service delivery is outsourced, only responsibility for service provision is outsourced — accountability remains with the organization
  • Assurance methods: questionnaires, onsite visits, third-party reports (SSAE 18 SOC 1, SOC 2, SOC 3)

4.11 Database Management

Overview

A DBMS organizes, controls, and enables use of data for application programs.

DBMS Functions: Reduced data redundancy, decreased access time, basic security over sensitive data.

DBMS Advantages: - Data independence for application systems - Flexibility in meeting changing data requirements - Transaction processing efficiency - Reduction of data redundancy - Maximized data consistency - Minimized maintenance cost through sharing - Enforced data/programming standards and security - Data integrity checks

4.11.1 DBMS Architecture

Metadata — Data elements defining the database (logical/physical fields, relationships, queries).

Three types of metadata/schemas:

Schema Type Description
Conceptual schema High-level logical view
External schema User/application view
Internal schema Physical storage details

Data Dictionary/Directory System (DD/DS): - Identifies fields along with characteristics and uses - Active data dictionary — Requires entries for all data elements; assists processing - Passive dictionary — Read-only repository - Benefits: Enhanced documentation, common validation, facilitates programming, standardized methods

4.11.2 Database Structure Types

Model Description Key Feature Use Case
Hierarchical Parent-child relationships; tree structure; 1:N Only one parent per child; data duplication needed for multi-parent Windows Registry, Geographic Info Systems
Network Set-based; owner + member records; allows multiple parents Complex, difficult to modify Rarely used in current environments
Relational Tables (rows/tuples + columns/attributes); SQL-based Dynamic; join operations; normalization; most flexible ERP systems; Oracle, DB2, SQL Server
Object-Oriented (OODBMS) Data stored as objects; encapsulation + inheritance Supports complex data types Engineering, science, spatial databases, graphics/audio
NoSQL Developed for big data; unstructured data Sharding, dynamic schemas MongoDB, Cassandra

Relational Database Properties: - Values are atomic (single, irreducible) - Each row uniquely identifiable (primary key) - Column values are of the same kind - Sequence of rows/columns is insignificant - Each column has a unique name

Normalization Rules (Relational): - Each attribute has only one value per instance - Attributes contain no internal structure - Each tuple = primary key + independent attributes - Foreign keys: null or linked to existing value (referential integrity)

NoSQL Database Types:

Type Description
Key-value Items stored as attribute name (key) with value
Column-oriented Values of a column grouped together
Graph database Based on graph theory (mathematical relationship models)
Document-oriented Manages document-oriented info (XML, JSON)
Time series Data with timestamps (stock data, logs, sensors)
Spatial Stores/queries spatial data (maps, GPS)

NoSQL Advantage: Sharding (horizontal partitioning across servers) and dynamic schemas.

4.11.3 Database Controls

Controls to maintain database integrity and availability: - Establish definition standards - Implement data backup and recovery procedures - Establish access controls (including privileged access) - Ensure only authorized personnel can update - Handle concurrent access (transaction commit, record locking) - Ensure accuracy, completeness, and consistency in table/column definitions - Minimize use of non-system tools - Use database checkpoints to minimize data loss - Implement DAM (Database Activity Monitoring) technology - Use performance reporting tools (storage space, buffer size, CPU use)

PART B: BUSINESS RESILIENCE

4.12 Business Impact Analysis (BIA)

Definition

BIA is a critical step in developing the business continuity strategy. It evaluates critical processes, determines timeframes, priorities, resources, and interdependencies.

BIA Requires: - High-level senior management support/sponsorship - Extensive IT and end-user personnel involvement - Approval of information resource criticality by senior management

Three Key BIA Questions:

1. What are the different business processes? Indicators of criticality: - Supports health and safety (hospital records, air traffic control) - Disruption causes income loss or exceptional costs - Required for legal/statutory compliance - Affects significant number of users/business segments

2. What critical information resources support critical business processes? Examples of critical processes: - Receiving payments, producing goods/services - Paying employees, advertising - Dispatching finished goods - Regulatory compliance

3. What is the critical recovery period? - Financial institutions → shorter recovery period - Manufacturing → longer recovery period - Time of year/week matters (e.g., Saturday midnight gives more recovery time than Monday midnight for a bank)

BIA Data Collection Methods:

  1. Questionnaire approach — Detailed questionnaire circulated to key users
  2. Interview approach — Group interviews with key users
  3. Workshop approach — IT + end users together to determine acceptable downtime and vital resources

Cost Analysis

Two Independent Cost Factors:

Total Cost = Downtime Cost + Recovery Cost Downtime Cost: Increases with time (idle resources, lost sales, financial costs, indirect costs)
Recovery Cost: Decreases with target recovery time (BCP preparation, testing, offsite backup, insurance) Optimal point = Minimum of Total Cost curve (U-curve bottom)

Classification of Operations (Criticality Analysis)

Classification Description Manual Alternative
Critical Cannot be performed unless replaced by identical capability; manual methods impossible None possible
Vital Can be performed manually but only briefly (usually ≤5 days) Short-term manual possible
Sensitive Can be performed manually at tolerable cost for extended period but difficult Extended manual possible with extra staff
Nonsensitive Interruption for extended period with little/no cost Easy manual alternative

Key Recovery Metrics

Metric Definition Implication
RPO (Recovery Point Objective) Earliest acceptable time to recover data; how much data loss is acceptable Low RPO → need data mirroring/real-time replication
RTO (Recovery Time Objective) Earliest time business operations must resume after disaster Low RTO → need hot site, clustering
MTTR (Mean Time to Repair) Average time to repair a failed system High MTTR → longer downtime, lost productivity

Exam Critical: RPO affects DATA PROTECTION solutions. RTO affects RECOVERY SITE/INFRASTRUCTURE solutions.

Additional Parameters: - Interruption window — Maximum wait time from failure to restoration - SDO (Service Delivery Objective) — Level of service during alternate mode - MTO (Maximum Tolerable Outage) — Maximum time in alternate mode before bigger problems arise

ALE (Annual Loss Expectancy): Risk × Impact = Maximum reasonable preparation cost Example: 0.1% risk × $10M impact = $10,000/5 years

4.13 System and Operational Resilience

System Resilience

Ability of a system to withstand/adapt to disruptions while maintaining core functions.

Built using: Redundant components, failover mechanisms, robust backup strategies.

Application Resiliency: Clustering

Cluster — Software installed on every server (node) in which application runs. Prevents single point of failure (SPOF).

Cluster Type Description Recovery Requirement
Active-Passive App runs on ONE active node; passive nodes take over on failure Brief downtime on failover No special app setup needed
Active-Active App runs on ALL nodes; load balanced; concurrent data access Near-zero downtime App must be cluster-aware

Geo-clusters — Span cities/countries/continents Metro-clusters — Span one city

Telecommunication Network Resiliency

Methods for Network Protection:

Method Description
Redundancy Extra capacity; multiple paths between routers; dynamic routing (OSPF, EIGRP); failover devices
Alternative routing Different medium (copper vs fiber); different networks if normal unavailable; counter-rotating fiber rings
Diverse routing Traffic through split or duplicate cable facilities with different sheaths
Long-haul network diversity Multiple long-distance carriers; automatic rerouting software
Last-mile circuit protection Redundant local carrier circuits + microwave + coaxial access to local loop
Voice recovery Redundant cabling and VoIP
Satellite connectivity Broadband satellite for locations lacking other options

4.14 Data Backup, Storage and Restoration

RAID (Redundant Array of Independent Disks)

Definition: Data storage virtualization combining multiple physical disks into logical units for redundancy/performance.

Data Replication Types:

Type Description Performance Impact
Synchronous Local write not confirmed until remote site confirms receipt Lower write performance; data safe
Asynchronous Data replicated on scheduled basis with some delay Better performance; some data loss possible
Adaptive Switches between modes based on network load Best of both worlds

4.14.2 Backup and Restoration

Backup Types:

Type What's Backed Up Restore Requirement Speed Storage
Full ALL files and folders One backup set Slowest Most
Incremental Only files changed since LAST incremental or full All incremental sets since last full Fastest to backup Least
Differential All files changed since LAST FULL backup Last full + last differential Medium Medium

Exam Point: Incremental = only changes since LAST incremental. Differential = all changes since LAST FULL backup (cumulative). Restoring differential is FASTER than incremental; backing up incremental is FASTER than differential.

Media Rotation: Grandfather-Father-Son (GFS) Method: - Son = Daily backups (within week) - Father = Last backup of the week - Grandfather = Last backup of the month (year-end → yearly backup)

3-2-1 Backup Strategy: - 3 copies of data (1 primary + 2 backups) - 2 different types of storage media - 1 copy offsite

Immutable Backup — Cannot be altered, changed, or deleted; protects against ransomware.

Offsite Library Controls: - Secure physical access (authorized personnel only) - Encrypt backup media (especially in transit) - Fire-resistant physical construction - Located away from data center (different disaster zone) - Maintain media inventory for specified retention time - Record all media movements

Cloud Backup Controls: - Encrypt data in transit - Store encrypted data; limit access to encryption keys - Verify backup/restore times meet RTO/RPO requirements

Types of Backup Devices:

Type Description Use Case
VTL (Virtual Tape Library) Disk storage mimicking tape library DR with hardware-based replication
Host-based replication Software on servers; real-time (sync) or delayed (async) Flexible; no specialized hardware
Disk-array replication Hardware-level replication; transparent to apps Highest performance
Snapshot technology Point-in-time copies; full or changed blocks only Efficient backup without production overhead

4.15 Business Continuity Plan (BCP)

Overview

BCP enables the business to continue operations during any kind of disturbance. It is primarily the responsibility of senior management.

BCP Components: - Continuity of operations plan — Maintain operations during recovery - DRP — Recover facility rendered inoperable; relocate operations - Business resumption plan — Return to normal operations

BCP Takes Into Consideration: - Critical operations necessary for survival - Human/material resources supporting them - Contractual commitments (including third-party IT services)

4.15.1 IT Business Continuity Planning

Key Principle: Application criticality is determined by the role of the application in supporting the organization's strategy — NOT by replacement cost, vendor support, or associated threats.

Minimizing Disaster Likelihood: - Data center location: Not on flood plain, earthquake fault line, or near explosive materials - Resilient network topologies (Loop or Mesh with alternative processing facilities built in)

4.15.2 Disasters and Disruptive Events

Types of Disruptions: - Natural: earthquakes, floods, tornados, fire, severe weather - Utility failures: power, telecommunications, gas supply - Human-caused: terrorist attacks, hacker attacks, human error - System malfunctions: accidental deletions, untested releases, DoS attacks, viruses

Pandemic Planning: Unique challenges — scale and duration hard to predict; impacts people differently than physical disasters.

Black Swan Events: Surprise events with major effects that cannot be anticipated; management should consider contingency planning for critical processes.

Image/Reputation Management: - Appoint trained spokesperson ONLY - Pre-draft generic announcement with blanks - Don't state causes; state investigation started - Don't assume liability - Don't blame the system

4.15.3 Business Continuity Planning Process (Life Cycle)

Initiation → Risk Assessment & Analysis → BIA → Strategy →
Execution (risk countermeasures) → BC Awareness Training → Plan Development & Testing → Maintenance

4.15.4 Business Continuity Policy

Purposes: - Message to internal stakeholders (employees, management, board) - Communication to external stakeholders (shareholders, regulators) - Empowers those responsible for BC - States general principles (risk assessment, BIA, redundancy targets)

Key Principle: BCP policy should be proactive — detect/prevent disruptions AND mitigate consequences.

4.15.5 Incident Classification

Level Description Example
Negligible No perceptible damage Brief OS crash with full recovery; momentary power outage with UPS
Minor No material financial impact Internet outage in single office of large corporation
Major Negative material impact; may affect other systems Ransomware attack compromising servers
Crisis Serious material impact on continued functioning; substantially disrupts operations Multi-site outage; data loss at scale

Starting point: Any non-negligible incident defaults to provisional severity Level 3; reassessed as incident evolves.

4.15.8 Components of a BCP (Figure 4.39)

Plan Purpose
BCP Procedures for sustaining mission/business operations during significant disruption
COOP (Continuity of Operations Plan) Guidance to sustain MEFs at alternate site for up to 30 days; federally mandated
Crisis Communications Plan Internal/external communications; control rumors
CIP (Critical Infrastructure Protection) Plan Protect national critical infrastructure
Cyber Incident Response Plan Mitigate/correct cyberattacks (virus, worm, Trojan)
DRP Procedures for relocating IS operations to alternate location
ISCP (Information System Contingency Plan) Recover single information system
OEP (Occupant Emergency Plan) Minimize loss of life/injury; protect property

4.15.9 Plan Testing

Test Types (in order of increasing rigor):

Test Type Description Disruption Risk
Desk-based/Tabletop (paper test) Walk through plan on paper with key players None
Preparedness test Localized version using actual resources; partial simulation Low
Full operational test Complete shutdown and full recovery execution High

Test Phases: 1. Pretest — Setup actions (transporting equipment, etc.) 2. Test — Actual operational activities 3. Posttest — Cleanup; evaluate plan; implement improvements

Test Measurements: - Time — Elapsed time for tasks - Amount — Work performed at backup site - Count — Vital records at backup site vs. required; critical systems recovered - Accuracy — Data entry accuracy vs. normal

Plan Maintenance Triggers: - Business process changes (new departments, key personnel) - New resources/applications acquired - Business strategy changes - Software/hardware environment changes - New threat events - Key personnel contact changes

BCP Coordinator Responsibilities: - Periodic review schedule (at least annually) - Unscheduled revisions for significant changes - Update plan within 30 days of review - Coordinate scheduled and unscheduled tests - Training on emergency/recovery procedures - Maintain historical documentation

4.15.11 Auditing Business Continuity

IS Auditor Tasks: - Understand BC strategy and connection to business objectives - Review BIA findings (reflect current priorities) - Evaluate BCPs against standards/regulations (including RTO/RPO) - Verify BCP effectiveness via previous test results - Evaluate cloud-based mechanisms - Evaluate offsite storage (facility inspection, contents, security, environmental) - Verify backup media transport security - Evaluate personnel emergency response capability - Ensure plan maintenance process is in place

IS Auditor Role in DRP Testing: IS auditor should observe tests — NOT develop, maintain, or declare disasters. IS auditor should ALWAYS be present during DRP tests to ensure procedures meet targets.

4.16 Disaster Recovery Plans (DRP)

Overview

DRP is an element of internal control to manage availability and restore critical processes/IT services after interruption.

Key Outcomes of DR Planning: 1. Changes to IT infrastructure, processes, procedures (IT DR strategies spanning 3-5 years) 2. DRPs documenting response to incidents from simple emergencies to full disasters

4.16.1 RPO, RTO, and MTTR

Exam Critical Table:

Metric Definition If LOW, Use... If HIGH, Use...
RPO Maximum acceptable data loss (in time) Data mirroring, real-time replication Tape backups, periodic backups
RTO Maximum acceptable downtime Hot site, clustering, dedicated spare servers Warm/cold site, manual recovery
MTTR Average repair time N/A (lower is better) N/A

Technology vs. RTO/RPO (Center = 0-1 hour, more expensive):

RPO (0→hours): Real-time replication → Delayed replication → Log shipping → Snapshots → Disk-based → Tape backups RTO (0→hours): Active-active clustering → Active-passive clustering → Hot standby → Cold standby

MTTR Factors: - Complexity of system - Availability of spare parts - Technician skill and experience - Number of technicians available - Maintenance procedures

4.16.3 Recovery Alternatives

Alternative Description Readiness Cost
Cold site Space + basic infrastructure; NO IT equipment Needs hardware/software installation when activated Lowest
Warm site Partially configured with network + some peripherals; NO main computer Programs/current data must be loaded Medium
Hot site Fully configured; all IT + communications equipment installed; may have near-real-time data Operational within hours/minutes High
Mirrored site Fully redundant; real-time data replication; fully staffed Immediate; no interruption Highest
Mobile site Packaged facility on transportable vehicles Delivered and set up at specified location Varies
Reciprocal agreements Agreement between similar companies to share IT facilities Difficult to enforce; compatibility issues; not recommended Low (but unreliable)

Sports Analogy: Cold site = substitute on bench; Warm site = substitute warming up; Hot site = substitute on sideline ready to enter.

Selection Factors: - Criticality of business process and supporting applications - Cost - Time required to recover - Security

Alternate Site Considerations: - Beyond geographic area affected by the same disaster - Adequate separation from primary site based on nature of disruptive events

4.16.4 Development of DRP

IT DRP Contents: - Procedures for declaring a disaster (escalation procedures) - Criteria for plan activation - Linkage with overarching plans (BCP, crisis management plan) - Recovery teams and their responsibilities - Contact and notification lists - Step-by-step recovery process - Recovery procedures for each IT system/component - Vendor and supplier contacts - Resources required for recovery

DRP Scenarios: - Loss of network connectivity - Loss of key IT system - Loss of processing site (server room) - Loss of critical data - Loss of office - Loss of key service provider (cloud)

Recovery Teams:

Team Responsibility
Incident response team Receives all incident reports; postmortem analysis
Emergency action team First responders, evacuation, fire wardens
Information security team Maintains security at alternate site; monitors security
Damage assessment team Assesses extent of damage; estimates recovery time
Emergency management team Coordinates all teams; key decisions; activates BCP
Offsite storage team Obtains/ships media and records to recovery facility
Software team Restores system packs, loads/tests OS software
Applications team Restores backup system/application programs
Network recovery team Reroutes communications; reestablishes host network control
Transportation team Locates recovery site; coordinates employee transport
Salvage team Detailed damage assessment; filing insurance claims
Relocation team Coordinates move from hot site to new/restored location
Legal affairs team Handles legal issues from incidents/service unavailability

4.16.5 Disaster Recovery Testing Methods

Types of Tests:

Test Type Description Disruption Level
Checklist review Distribute checklists to team members; review currency None
Structured walk-through Team physically implements plans on paper; assess effectiveness None
Simulation test Roleplay disaster scenario; do NOT activate recovery site None
Parallel test Recovery site brought to operational readiness; PRIMARY SITE CONTINUES normally Low
Full interruption test Operations SHUT DOWN at primary; shifted to recovery site; most rigorous High

Testing Progression: Start simple → increase gradually → build confidence → minimize risk.

Surprise tests — Advantageous (realistic) but potentially disruptive and can alienate personnel.

Test Metrics: - Time (elapsed time for tasks; was RTO met?) - Data (all required data recovered; was RPO met?) - Amount (work performed at backup site) - Count (critical systems recovered; transactions processed) - Accuracy (data entry accuracy vs. normal)

4.16.6 Invoking DRP

Triggering Process: 1. Designated individual notified immediately when triggering event occurs 2. Follow pre-established escalation protocol 3. Invoke appropriate recovery plan (IT DRP) 4. Mobilize required teams 5. Evaluate incident to match against tested scenario 6. Follow documented recovery procedures 7. Document all decisions made → update procedures after recovery

PROBABLE EXAM QUESTIONS

Q1: Which recovery site is MOST appropriate for mission-critical systems with low tolerance for interruption? Answer: Hot site — fully configured, ready within hours/minutes.

Q2: An IS auditor is testing change management. What is the MOST effective method? Answer: Trace from system-generated information to change management documentation (to detect undocumented changes). Going the other direction risks missing unauthorized changes.

Q3: Data mirroring should be implemented when: Answer: RPO is low (minutes) — organization cannot afford to lose even a few minutes of data.

Q4: The criticality classification of a software application in BCP is determined by: Answer: The nature of the business and the value of the application to the business (its role in supporting business strategy).

Q5: When auditing client-server database security, the IS auditor should be MOST concerned about: Answer: System utilities — they may enable unauthorized changes to data without adequate controls.

Q6: The FIRST thing an IS auditor should examine when reviewing a network for Internet communications: Answer: Network architecture and design — provides the overall picture of network connectivity.

Q7: Which allows an enterprise to extend its intranet across the Internet to business partners? Answer: VPN (Virtual Private Network) — uses tunneling/encapsulation over public networks.

Q8: Which component of BCP is PRIMARILY the IT department's responsibility? Answer: Restoring IT systems and data after a disaster within designated timeframes.

Q9: Who is PRIMARILY responsible for developing the BCP? Answer: Senior management — they are accountable for the results.

Q10: What is benchmarking used for in IT performance evaluation? Answer: Determining the level of performance offered by similar IPF environments (not user satisfaction or goal accomplishment).

Q11: Which backup type requires the LEAST storage but the LONGEST restore time? Answer: Incremental backup.

Q12: The purpose of SNMP is: Answer: To monitor and control network variables, manage configurations, and collect performance/security statistics across all network devices.

Q13: An IS auditor should be involved in DRP testing by: Answer: Observing the test — NOT developing, maintaining, or executing the DRP.

Q14: Shadow IT refers to: Answer: Use of systems/services/hardware/software without proper IT/cybersecurity department approval.

Q15: What is the relationship between RTO and disaster tolerance? Answer: The lower the RTO, the lower the disaster tolerance — the business cannot tolerate prolonged downtime.
